# 貸款模組解析

本文整理貸款模組的程式碼結構、功能流程、資料模型、前後端對接、跨模組整合與建議閱讀順序。內容聚焦在程式碼與功能本身，不記錄分支比較過程或更新狀態。

## 1. 模組定位

貸款模組負責從「客戶送出貸款申請」到「行員審核、風控判斷、帳戶撥款、客戶還款、貸款結清」的完整生命週期。

它不是單一獨立模組，而是會和多個模組互動：

| 關聯模組 | 貸款模組使用方式 |
|---|---|
| Customer | 取得客戶姓名、CIF、電話、Email 等基本資料 |
| Account | 建立貸款帳務帳戶、撥款、還款、查可扣款帳戶、查交易紀錄 |
| Risk | 送出貸款審核、補件後通知風控、接收風控 callback |
| Common | 使用 Security、JWT、Email、檔案儲存、RestTemplate、例外處理與 PDF 中文字型載入工具 |
| Notification / Mail | 申請成立、補件、拒絕、核准撥款、契約附件、還款成功、逾期、結清等通知 |
| Frontend | 提供客戶端貸款頁與行員端貸款管理頁 |

## 2. 主要使用者與功能

### 2.1 客戶端

| 功能 | 前端畫面 | 後端入口 | 說明 |
|---|---|---|---|
| 申請貸款 | `LoanApplyView.vue` | `LoanApplicationController` | 客戶選擇貸款類型、金額、期數、撥款帳戶後送出申請 |
| 查詢申請狀態 | `LoanStatusView.vue` | `LoanApplicationController` | 客戶查看自己的申請進度、聯繫狀態、核准條件 |
| 上傳補件 | `LoanStatusView.vue` | `LoanDocumentController` | 申請需要補件時，上傳身分證、收入證明等文件 |
| 查看貸款帳戶 | `LoanAccountView.vue` | `LoanAccountController` | 撥款完成後查看貸款帳戶、本金、利率、期數、還款進度 |
| 執行還款 | `LoanRepaymentView.vue` | `AccountIntegrationController` | 選擇扣款帳戶並繳納本期或多期款項 |
| 查還款紀錄 | `LoanRepaymentView.vue` | `AccountIntegrationController` | 查看貸款還款交易紀錄 |

客戶端限制：

1. 客戶只能查詢與操作自己的貸款申請、補件與貸款帳戶。
2. 客戶端身分主要由 JWT 中的 `customerId` 判斷。
3. 撥款帳戶與扣款帳戶都必須是本人名下可用帳戶。
4. 補件正式送出後不可任意刪除。
5. 還款金額必須符合本期或連續多期應繳金額，最後一期可依剩餘本金清償。

### 2.2 行員端

| 功能 | 前端畫面 | 後端入口 | 說明 |
|---|---|---|---|
| 查申請列表 | `LoanApplicationView.vue` | `LoanAdminController` | 依狀態、類型、分頁與排序查看申請 |
| 新增聯繫紀錄 | `LoanContactLogModal.vue` | `LoanAdminController` | 記錄電話、Email、SMS 等聯繫結果 |
| 查看補件 | `LoanDocumentModal.vue` | `LoanDocumentController` | 查看客戶已送出的文件 |
| 二次填單 | `LoanReviewModal.vue` | `LoanAdminController` | 填核准金額、期數、利率與擔保備註 |
| 正式送審 | `LoanReviewModal.vue` | `LoanAdminController` | 將審核資料送往風控 |
| 重送風控 | `LoanApplicationView.vue` | `LoanAdminController` | 風控未收到或狀態卡住時補償重送 |
| 重試撥款 | `LoanApplicationView.vue` | `LoanAdminController` | 核准後撥款卡住時補償觸發 |
| 查貸款帳戶 | `LoanAccountAdminView.vue` | `LoanAccountAdminController` | 查看所有貸款帳戶、篩選狀態、類型與帳號 |
| 查還款時間表 | `LoanAccountAdminView.vue` | `LoanAccountAdminController` | 查看指定貸款帳戶的所有期數 |
| 手動同步已繳 | 後台 API | `LoanAccountAdminController` | 帳務已扣款但貸款期數需補同步時使用 |

行員端重點：

1. 後台申請列表會直接顯示客戶姓名與聯絡資訊，方便行員聯繫。
2. 聯繫紀錄是 append-only 思路，保留歷史互動。
3. 二次填單可先存草稿，再正式送審。
4. 後台貸款帳戶頁支援用案件編號與貸款帳務帳號搜尋。
5. 撥款與風控都有補償 API，避免跨模組流程因外部呼叫失敗而卡住。

## 3. 核心流程

### 3.1 客戶申請流程

```text
LoanApplyView
-> GET /api/loan-applications/rate-rules
-> 選貸款類型、金額、期數、撥款帳戶
-> POST /api/loan-applications/member
-> LoanApplicationController
-> LoanApplicationService.insertMember
-> LoanApplication
-> EmailService.sendLoanAppliedNotification
-> 狀態 PENDING_CONTACT
```

這段流程負責建立申請主檔。前端會先載入利率規則，讓客戶可以試算月付金；送出後後端建立 `LoanApplication`，並寄送申請成立通知。

### 3.2 行員聯繫與審核流程

```text
LoanApplicationView
-> LoanAdminController.getByStatus
-> LoanContactLogModal
-> LoanAdminController.addContactLog
-> LoanReviewModal
-> LoanAdminController.saveReviewDetail
-> LoanAdminController.submitReview
-> LoanApplicationService.submitReview
```

這段流程負責把客戶申請轉成可送風控的審核資料。行員可以新增聯繫紀錄，也可以填入核准金額、期數、利率、擔保備註。

### 3.3 風控送審與 callback

```text
LoanApplicationService.submitReview
-> transaction afterCommit
-> LoanRiskClient.submitForReview
-> Risk module
-> POST /api/loan-callbacks/{applicationId}/status
-> LoanCallbackController
-> LoanApplicationService.handleStatusCallback
```

送風控使用 afterCommit，是為了避免資料庫還沒提交就先呼叫外部模組。風控回傳後，貸款模組會依結果轉成核准、拒絕或補件狀態。

### 3.4 補件流程

```text
LoanStatusView
-> LoanDocumentController.upload
-> LoanDocumentService.upload
-> FileStorageService
-> LoanDocument
-> LoanDocumentController.submitDocuments
-> LoanDocumentService.submitDocuments
-> LoanRiskClient.attachDocuments
```

補件流程負責保存檔案與文件後設資料。檔案本體交給 `FileStorageService`，資料表保存文件類型、原始檔名、儲存路徑、上傳時間等資訊。

### 3.5 核准後建帳與撥款

```text
Risk callback APPROVED
-> LoanApplicationService.handleStatusCallback
-> afterCommit
-> LoanApplicationService.autoDisburse
-> AccountIntegrationService
-> 建立 AccountType.LOAN 帳戶
-> 撥款到客戶指定入帳帳戶
-> Account callback
-> LoanAccountService.createFromDisbursement
-> LoanRepaymentService.createSchedule
-> LoanContractPdfService.generateContractPdf
-> EmailService.sendLoanContractEmail
```

核准後會呼叫帳戶模組建立貸款帳務帳戶，並將核准本金撥入客戶指定帳戶。帳戶模組回調後，貸款模組建立 `LoanAccount` 與每一期 `LoanRepayment`，接著產生貸款契約 PDF，透過 `EmailService.sendLoanContractEmail` 寄送給客戶。

### 3.6 還款流程

```text
LoanRepaymentView
-> GET /api/loan-accounts/my
-> GET /api/customer/loan-repayments/debit-accounts
-> POST /api/customer/loan-repayments
-> AccountIntegrationService.repayLoan
-> 帳務扣款與交易紀錄
-> LoanRepaymentService.processRepayments
-> 更新期數與貸款帳戶狀態
```

還款會先由帳戶模組處理扣款與交易紀錄，再由貸款模組同步還款期數。若全部清償，貸款帳戶會轉成 `PAID_OFF`，申請可進入結清狀態。

### 3.7 排程流程

```text
LoanRepaymentScheduler
-> 每日 01:00
-> 查詢逾期或即將到期的 LoanRepayment
-> 更新逾期狀態
-> EmailService 寄送逾期或提醒通知
```

排程負責日常維護，不需要人工操作。它會檢查逾期與即將到期的還款期數，並發送通知。

## 4. 後端檔案職責

### 4.1 Controller

| 檔案 | 職責 |
|---|---|
| `LoanApplicationController.java` | 客戶端貸款申請入口，提供申請、查自己的申請、查利率規則 |
| `LoanAdminController.java` | 行員端申請管理入口，提供申請列表、聯繫紀錄、二次填單、送審、補償重送 |
| `LoanDocumentController.java` | 補件文件入口，提供客戶上傳、送出、刪除，以及行員查詢 |
| `LoanAccountController.java` | 客戶端貸款帳戶入口，提供自己的貸款帳戶與還款表 |
| `LoanAccountAdminController.java` | 行員端貸款帳戶入口，提供全部帳戶、還款表、手動同步 |
| `LoanCallbackController.java` | 外部 callback 入口，接收風控與帳戶模組回傳 |

## 5. 資料模型

### 5.1 Entity

| Entity | 資料表 | 用途 |
|---|---|---|
| `LoanApplication` | `loan_application` | 貸款申請主資料，包含客戶、貸款類型、申請金額、期數、利率、狀態、撥款帳號 |
| `LoanContactLog` | `loan_contact_log` | 行員聯繫紀錄 |
| `LoanReviewDetail` | `loan_review_detail` | 行員二次填單與送審資料 |
| `LoanDocument` | `loan_document` | 補件文件後設資料，包含批次類型、批次編號與正式送出時間 |
| `LoanAccount` | `loan_account` | 撥款後的貸款帳戶資料，包含帳務帳號、剩餘本金、下一期日期、帳戶狀態 |
| `LoanRepayment` | `loan_repayment` | 每一期還款資料，包含本金、利息、應繳日、已繳狀態 |

### 5.2 LoanAccount 欄位特色

`LoanAccount` 目前明確使用 `@Column` 對應資料表欄位，例如：

| 欄位 | 說明 |
|---|---|
| `accountId` | 貸款模組內部主鍵 |
| `accountNumber` | 帳戶模組產生的貸款帳務帳號 |
| `applicationId` | 對應原始貸款申請 |
| `customerId` | 客戶識別 |
| `applyType` | 貸款類型 |
| `principalAmount` | 核准本金 |
| `confirmedPeriod` | 核准期數 |
| `rate` | 年利率 |
| `monthlyPayment` | 每期應繳金額 |
| `paidPeriods` | 已繳期數 |
| `remainingPrincipal` | 剩餘本金 |
| `nextPaymentDate` | 下一期繳款日 |
| `accountStatus` | 貸款帳戶狀態 |

這個 entity 是貸款帳戶頁與還款流程的核心資料。

### 5.3 DTO

| DTO | 用途 |
|---|---|
| `LoanMemberRequestDTO` | 客戶送出貸款申請 |
| `LoanContactLogRequestDTO` | 行員新增聯繫紀錄 |
| `LoanReviewDetailRequestDTO` | 行員二次填單 |
| `LoanRiskRequestDTO` | 送風控審核資料 |
| `LoanStatusCallbackRequestDTO` | 風控或帳戶回調狀態 |
| `LoanDocumentInfoDTO` | 補件通知風控的文件摘要 |
| `LoanApplicationResponseDTO` | 回傳申請資料，包含客戶姓名、電話、Email 等行員需要資訊 |
| `LoanAccountResponseDTO` | 回傳貸款帳戶資料 |
| `LoanRepaymentResponseDTO` | 回傳還款期數 |
| `LoanDocumentResponseDTO` | 回傳補件文件 |

補件相關 DTO 與 Entity 會帶 `documentBatchType`、`documentBatchNo`、`submittedAt`。閱讀時可把它理解為「同一筆貸款申請可能有多輪補件」：初始補件與風控要求的後續補件會用批次欄位區分，前端只顯示目前應處理的那一批。

### 5.4 Enum

| Enum | 用途 |
|---|---|
| `LoanApplicationStatus` | 申請狀態，例如待聯繫、待審、核准、拒絕、補件、結清 |
| `LoanAccountStatus` | 貸款帳戶狀態，例如正常、逾期、結清 |
| `LoanRepaymentStatus` | 單期還款狀態，例如未繳、已繳、逾期 |

## 8. 風控整合

`LoanRiskClient` 負責 server-to-server 呼叫風控。

| 動作 | 說明 |
|---|---|
| 送審 | 行員正式送審後，把申請資料與審核資料送到風控 |
| 補件通知 | 客戶補件送出後，通知風控可以重新審查 |
| callback URL | 貸款模組提供 `/api/loan-callbacks/{applicationId}/status` 讓風控回傳 |
| 失敗處理 | 送審失敗會阻止狀態進入等待風控，避免資料卡死 |

風控回傳可能導致：

1. 申請核准，進入自動撥款。
2. 申請拒絕，寄送拒絕通知。
3. 要求補件，客戶端開放補件流程。

## 9. 文件與通知

### 9.1 補件文件

補件由 `LoanDocumentController` 與 `LoanDocumentService` 處理。

```text
前端 FormData
-> MultipartFile
-> LoanDocumentService
-> FileStorageService
-> LoanDocumentRepository
```

文件類型由 `LoanDocumentType` 控制，資料表只保存檔案後設資料，檔案本體由 `FileStorageService` 落地。

### 9.2 Email

貸款相關 Email 集中在 `EmailService`：

| 通知 | 時機 |
|---|---|
| 申請成立 | 客戶送出貸款申請 |
| 補件通知 | 風控或行員要求補件 |
| 拒絕通知 | 申請被拒絕 |
| 核准撥款 | 貸款核准且撥款完成 |
| 貸款契約 | 撥款完成後，將契約 PDF 作為附件寄給客戶 |
| 還款成功 | 客戶完成還款 |
| 結清通知 | 全部貸款清償 |
| 逾期通知 | 排程掃描到逾期 |
| 到期提醒 | 還款日前 1 至 3 天提醒 |

### 9.3 PDF

`LoanContractPdfService` 產生貸款契約 PDF，並使用 Common 的 `PdfCjkFontLoader` 從 classpath 載入 `src/main/resources/fonts/NotoSansTC-Regular.ttf`。這樣做的目的，是避免 PDF 服務依賴本機作業系統字型路徑，讓開發機與部署環境都能穩定顯示中文。產生完成後，PDF 會作為貸款契約通知信附件寄出。

## 14. 維護提醒

1. 貸款是跨模組流程，改貸款狀態時要同步確認風控、帳戶、通知與前端顯示。
2. 金額與利率請維持 `BigDecimal`，不要改用浮點數。
3. 送風控與撥款這類外部副作用，應避免發生在 DB transaction 還沒提交前。
4. 補件檔案刪除時要同時考慮資料表紀錄與實體檔案。
5. 還款流程需避免重複扣款、重複標記已繳與重複結清。
6. 後台貸款帳戶頁顯示的是貸款模組帳戶資料，但帳務帳號來自 Account 模組。
7. 申請列表若要顯示客戶聯絡資訊，要確認 DTO 與 `LoanApplicationService` 有從 `CustomerProfile` 補齊。
8. 契約 PDF 與寄信流程橫跨 `LoanApplicationService`、`LoanContractPdfService`、`EmailService`、`loan-contract.html` 與 `NotoSansTC-Regular.ttf`，改其中一段時要一起確認。
9. mock data 若調整貸款本金、剩餘本金或帳務負債，需同步確認 Account 模組帳戶負債金額，避免畫面與帳務資料不一致。
